## FINDING 1: Pandoc Native Template Language (`$var$` syntax)

### Finding Details
Pandoc's built-in template language uses `$var$` syntax and provides:
- **Variable interpolation**: `$title$`, `$author$`, `$lang$`
- **Conditionals**: `$if(var)$ ... $endif$` for simple boolean checks
- **Iteration**: `$for(items)$ ... $endfor$` with `$sep$` separators
- **Automatic metadata access**: All YAML front matter becomes template variables
- **Computed variables**: `$body$`, `$toc$`, `$date-meta$` generated by pandoc

### Strengths
✅ **No external dependencies** - Built into pandoc  
✅ **Simple and readable** - Easy to understand template logic  
✅ **Zero overhead** - Native implementation  
✅ **RTL/LTR ready** - Your current template already uses `$if(lang)$ lang="$lang$"$endif$`  
✅ **Well-documented** - Official pandoc documentation covers it thoroughly  
✅ **Battle-tested** - Used by thousands of documentation projects  

### Limitations
❌ **No arithmetic** - Can't do calculations or string manipulation  
❌ **No custom functions** - Can't define reusable logic  
❌ **No complex filtering** - Limited to simple boolean checks  
❌ **Loops are basic** - No indexed loops, can't access loop index  
❌ **No nested data access** - Can't drill into complex object structures  

### Evidence
Your current template demonstrates this:
```html
<!-- ✅ Works: Variable interpolation -->
<html$if(lang)$ lang="$lang$"$endif$$if(dir)$ dir="$dir$"$endif$>

<!-- ✅ Works: Conditional with metadata loop -->
$if(keywords)$
  <meta name="keywords" content="$for(keywords)$$keywords$$sep$, $endfor$" />
$endif$

<!-- ✅ Works: Nested conditionals -->
<title>$if(title-prefix)$$title-prefix$ – $endif$$pagetitle$</title>
```

### Verified RTL/LTR Support
- **Tested**: `lang="fa" dir="rtl"` correctly appears in HTML output
- **Standard-compliant**: HTML5 spec for language/direction attributes
- **CSS integration**: Can use `html[lang="fa"]` selectors in CSS
- **Browser support**: All modern browsers understand lang/dir attributes

### Relevance to Global Objective
Your current setup is already optimized for this. The template language is sufficient for a blog with per-post metadata (title, author, date, language, direction). You don't need more power here unless you're generating complex layouts from markdown.

### Confidence
**HIGH** - This is well-documented in official pandoc manual, tested directly in your environment, and battle-tested by the community.

### References
- Pandoc Manual: https://pandoc.org/MANUAL.html#templates
- Your working template: `./static/_template.html`
- Testing: Confirmed with pandoc 3.9.0.2 in your environment

## FINDING 2: Pandoc Lua Filters - Powerful Content & Metadata Transformation

### Finding Details
Lua filters are small Lua scripts that run after pandoc parses the input and before rendering the output. They can:
- **Access all metadata** - Read/modify YAML front matter
- **Transform content blocks** - Header, Para, Div, CodeBlock, etc.
- **Transform inline content** - Str, Code, Link, Image, etc.
- **Modify document structure** - Add, remove, or change elements
- **Chain multiple filters** - Use `--lua-filter filter1.lua --lua-filter filter2.lua`
- **Access full pandoc AST** - Complete representation of the document

### Practical Capabilities

**1. Metadata Enhancement**
```lua
function Meta(meta)
  -- Auto-set direction based on language
  if meta.lang and pandoc.utils.stringify(meta.lang) == "fa" then
    meta.dir = "rtl"
    meta.body_class = "lang-fa"
  else
    meta.dir = "ltr"
    meta.body_class = "lang-en"
  end
  return meta
end
```
This enables: Auto-correction of lang/dir, computed fields, validation.

**2. Language-Aware Content Filtering**
```lua
function Div(div)
  local meta = PANDOC_DOCUMENT.meta
  local lang = pandoc.utils.stringify(meta.lang)
  
  -- Remove English-only content for Persian pages
  if div.classes:includes("only-en") and lang == "fa" then
    return {}  -- Remove block entirely
  end
  return div
end
```
This enables: Separate content paths per language in same markdown file.

**3. Header Enhancement**
```lua
function Header(header)
  -- Add language-specific classes
  if header.level == 1 then
    header.attributes.class = "main-heading"
    header.attributes["data-lang"] = "fa"  -- or detected from meta
  end
  return header
end
```
This enables: Language-aware styling, semantic HTML attributes.

### What Lua Filters CANNOT Do
❌ **Cannot override template** - Still uses your `_template.html`  
❌ **Cannot change output format** - Still generates HTML (or whatever you specified)  
❌ **Cannot generate arbitrary HTML** - Limited to AST transformations  
❌ **Cannot completely replace template rendering** - That requires Custom Writers  

### Strengths
✅ **No additional dependencies** - Lua is built into pandoc 2.0+  
✅ **Composable** - Chain multiple filters for different concerns  
✅ **Full access to content** - Can inspect and modify everything  
✅ **Excellent for bilingual** - Can handle language-specific content blocks  
✅ **Minimal performance impact** - Lua is fast  
✅ **Well-documented** - Official pandoc Lua filters guide  

### Limitations
❌ **Still uses template** - Can't completely override HTML structure  
❌ **Requires Lua knowledge** - Not as simple as template syntax  
❌ **Testing can be tricky** - Need to understand pandoc AST  

### Tested Implementations
Your environment confirmed:
- Metadata filters work correctly
- Can access and modify YAML front matter
- Filters can be chained successfully
- Lua 5.4 is available (from `pandoc --version`)

### Relevance to Global Objective
**HIGHLY RELEVANT** for your bilingual blog:
1. **Language validation** - Ensure lang/dir consistency
2. **Content separation** - Use `:::: only-en` and `:::: only-fa` divs
3. **Dynamic metadata** - Add computed fields like `is_rtl`, `lang_display_name`
4. **CSS integration** - Add classes and data attributes for styling

### Confidence
**HIGH** - Lua filters are official pandoc feature, tested in your environment, widely used in production.

### References
- Pandoc Lua Filters: https://pandoc.org/lua-filters.html
- Example filters tested: bilingual_lang_filter.lua, language_aware_filter.lua, metadata_validator_filter.lua
- Pandoc version: 3.9.0.2 (has Lua support enabled: `+lua`)

## FINDING 3: Pre-Processing with Template Engines (Jinja2/Tera/Mustache/Nunjucks)

### Finding Details
This approach processes markdown through a separate template engine BEFORE pandoc:
```bash
jinja2 -D var1=val1 input.md.j2 | pandoc --template _template.html
```

### Capabilities
✅ **Full template language** - if/for/filters/functions  
✅ **String manipulation** - substr, concat, filters  
✅ **Arithmetic** - Math operations  
✅ **Custom functions** - Define reusable logic  
✅ **Complex conditionals** - Full Turing-complete language  
✅ **Markdown generation** - Generate markdown content dynamically  

### Implementation Trade-offs

**Two-stage pipeline:**
```
Markdown template (input.md.j2)
  ↓ [Process with Jinja2/Tera/etc]
Markdown output (input.md)
  ↓ [Process with Pandoc]
HTML output (input.html)
```

**Advantages:**
- More powerful templating
- Can generate entire markdown structures dynamically
- Template engine features before pandoc processing

**Disadvantages:**
- Extra build step and tool dependency
- Overhead of markdown generation → markdown parsing → HTML
- More complex Justfile recipes
- Harder to debug (is the issue in template stage or pandoc stage?)

### Real-World Scenarios
**GOOD FOR:**
- Generating complex post structures from data
- Dynamic index pages with filtering/sorting
- Template-heavy static sites (like Hugo)

**NOT IDEAL FOR:**
- Simple blog with manual post writing
- When Lua filters are sufficient
- Adding external dependencies you don't need

### Example Justfile Integration
```bash
build FILE:
    #!/usr/bin/env bash
    input_file="$(basename "$(realpath "{{FILE}}")")"
    j2 -D lang=en -D year=2026 \
       "{{FILE}}" | \
    pandoc --template ./static/_template.html \
           --lua-filter ./filters/lang-aware.lua \
           -f markdown+smart -t html5 \
           -o "${input_dir}/${file_name}.html"
```

### Tested Observations
- Jinja2 CLI tools not installed in test environment
- Alternative tools: envsubst (shell variable substitution) works easily
- Python/Node/Rust equivalents all available (j2, nunjucks, tera)

### Relevance to Global Objective
**MODERATE** - Only if you have complex content generation needs. For a blog where posts are manually written in markdown, the overhead isn't justified.

### Confidence
**MEDIUM-HIGH** - The concept is sound and used in production static site generators. However, added complexity for your use case makes it not the first choice.

### References
- Jinja2 documentation: https://jinja.palletsprojects.com
- Tera template engine: https://tera.netlify.app
- Mustache: https://mustache.github.io
- Nunjucks: https://mozilla.github.io/nunjucks

## FINDING 4: Custom Lua Writers - Complete Control, High Complexity

### Finding Details
A Custom Lua Writer is a complete Lua script that generates the output format from scratch. It receives the full pandoc AST and produces output.

### Capabilities
✅ **Complete control** - Define entire HTML structure  
✅ **Full Lua language** - Any logic you want  
✅ **Novel formats** - Can generate any output format  
✅ **Metadata access** - Full document metadata available  

### Limitations
❌ **Cannot use templates** - Must define everything in Lua  
❌ **Much more code** - Have to handle all edge cases  
❌ **Complex to debug** - Large monolithic Lua script  
❌ **Harder to maintain** - Future pandoc AST changes require updates  
❌ **No easy composition** - Can't easily reuse logic from filters  

### Implementation Example (Conceptual)
```lua
function writeHtml(doc)
  -- Must write entire HTML structure manually
  -- Handle headers, paragraphs, lists, code blocks, etc.
  -- Generate from scratch, not using a template
end
```

### Testing Results
- Pandoc 3.9.0.2 in your environment doesn't use `--lua-writer` flag
- Custom writers are invoked with `-t custom_writer_name` syntax
- Would require significant Lua code to implement

### Verdict for Your Use Case
**NOT RECOMMENDED** - Lua filters already give you what you need without the complexity of writing a full custom writer.

### Relevance to Global Objective
**LOW** - Overkill for a bilingual blog. Lua filters are simpler and sufficient.

### Confidence
**HIGH** - This is well-documented in pandoc, but clearly not the right tool for your need.

### References
- Pandoc Custom Writers: https://pandoc.org/lua-filters.html#custom-writers
- Creating Output Formats: https://pandoc.org/extending-pandoc-with-lua.html

### Immediate Actions (Next Steps)

**Phase 1: Validate current setup (5 min)**
- ✅ Already done - Your template handles lang/dir correctly
- ✅ Already done - RTL/LTR working in HTML output

**Phase 2: Optional enhancements with Lua filters (if needed)**

**When to add Filter #1 (Language validation):**
If you want automatic dir assignment based on lang:
```bash
pandoc input.md \
  --lua-filter ./filters/lang-validator.lua \
  --template ./static/_template.html \
```

**When to add Filter #2 (Language-aware content):**
If you want English-only or Persian-only content blocks in the same markdown:
```markdown

::: only-en

```

Then use the language_aware_filter.lua to remove inappropriate blocks.

**Phase 3: Justfile integration**
Update your Justfile to include Lua filters:
```bash
build FILE:
    #!/usr/bin/env bash
    set -euo pipefail
    input_file="$(basename "$(realpath "{{FILE}}")")"
    input_dir="$(dirname "$(realpath "{{FILE}}")")"
    file_name="${input_file%.*}"
    
    disqus_opt=""
    if [[ "$input_file" != index*.md && "$input_file" != "README.md" ]]; then
        disqus_opt="--include-after-body=./static/disqus.html"
    fi
    
    pandoc \
        --template ./static/_template.html \
        --lua-filter ./filters/lang-validator.lua \
        --syntax-highlighting tango \
        -f markdown+smart -t html5 \
        --metadata date-meta="$(date +%Y-%m-%d)" \
        $disqus_opt \
        -o "${input_dir}/${file_name}.html" \
        "${input_dir}/${input_file}"
```

### Why NOT Other Approaches

**❌ Jinja2/Template preprocessing:**
- Adds build complexity for your use case
- You'd need another tool (jinja2, j2, tera, etc.)
- Markdown → generate markdown → parse markdown → HTML is inefficient
- Better suited for data-driven sites (not human-written blogs)

**❌ Custom Lua Writers:**
- Overkill - You don't need complete HTML control
- More code to maintain
- Lua filters already give you what you need
- Custom writers are for novel output formats, not blogs

**✅ Keep native template + add filters:**
- Minimal cognitive overhead
- Uses official pandoc features
- No new dependencies
- Follows community best practices
- Scales from simple to moderately complex
